## 1. 기능 개요

### 1.1 한 줄 요약

> 이 기능이 무엇을 하는지 한 문장으로 기술한다.

{예: Phase 7 통계 기반 3종 패턴 탐지 결과를 산출하고 `anomaly_results` 테이블에 적재한다.}

### 1.2 데이터 흐름

> 입력 → 처리 → 출력을 한 줄로 표현한다.
> AI가 코드를 어느 계층·파일에 작성해야 할지 판단하는 기준이 된다.

```
{예 — 파이프라인}
phase6/breakpoints/{cid}_{seg}_breakpoints.json + phase4/baseline/{cid}_{seg}_baseline.json
  → Phase 7 롤링 Z-score·IQR·패턴 1·2·3 판정
  → phase7/anomaly_events/{cid}_{seg}_anomaly_events.csv
  → anomaly_results 테이블 UPSERT

{예 — 백엔드}
anomaly_results + stat_timeseries 테이블
  → SQLAlchemy 쿼리 (commodity_id, segment_id, period 필터)
  → Pydantic 직렬화 (DATE → YYYY-MM 변환)
  → GET /api/v1/commodities/{id}/stream JSON 응답

{예 — 프론트엔드}
GET /api/v1/commodities/{id}/stream API 응답
  → Zustand useAppStore (selectedCommodityId, dateRange, filterState)
  → D3.js StreamChart 컴포넌트 렌더링 → 이상 노드 클릭 시 selectedAnomalyId 저장
```

### 1.3 프레임 내 위치

> 이 기능이 프레임에서 만들어진 어느 파일에 코드를 추가·수정하는지,
> 또는 새 파일을 어디에 생성하는지 명시한다.
> `frame_spec_vN.md §2 디렉토리 구조` 기준으로 경로를 작성한다.

| 구분 | 경로 | 작업 내용 |
|------|------|-----------|
| 수정 | `{기존 파일 경로}` | {추가·수정 내용 요약} |
| 신규 | `{새 파일 경로}` | {생성 이유} |

### 1.4 구현 범위 및 비구현 범위

| 구분 | 내용 |
|------|------|
| **구현** | {이번 브랜치에서 완성할 기능 목록} |
| **비구현** | {이번 브랜치에서 하지 않는 것 — 오해 방지용} |
| **선행 조건** | {이 브랜치 시작 전 완료되어 있어야 하는 브랜치·파일·결과} |

### 3.2 DB 적재 대상

> 해당 없는 경우 이 섹션을 삭제한다.

**적재 방식 정의**
- **INSERT**: 신규 행만 삽입 (중복 시 에러)
- **UPSERT**: UNIQUE 키 충돌 시 갱신 (표준 경로)
- **UPSERT (FK 후보정)**: FK 참조 대상 미존재 시 NULL 적재 후 후속 배치 보정

| DB 테이블 | 적재 컬럼 | UNIQUE 키 | 필터 조건 | 적재 방식 | 트랜잭션 단위 |
|-----------|-----------|-----------|-----------|-----------|---------------|
| `{테이블명}` | `{컬럼명}` | `({키 목록})` | {예: `confidence_grade IS NOT NULL`만 적재} | UPSERT | Phase {N} 단일 트랜잭션 |

### 3.3 API 응답

> 백엔드(API-\*) 기능이 아닌 경우 이 섹션을 삭제한다.
> 엔드포인트·필드명은 `api_spec_vN.md §{섹션}` 기준으로 명시한다.

| 엔드포인트 | 주 참조 테이블 | 응답 필드 | 비고 |
|------------|---------------|-----------|------|
| `GET /api/v1/{경로}` | `{테이블명}.{컬럼명}` | `{응답 JSON 키}` | {캐싱·페이지네이션·에러 코드 등} |

## 4. 파라미터 제약 조건

| {추가 파라미터} | `{키명}` | {기본값} | {이유} |

## 5. 예외처리

> - **`exception_spec_vN.md`**: 에러 코드 인덱스. 이 기능에 해당하는 코드의 발생 조건·처리 방침 확인 시 참조한다. (반복 조회용)
> - **`exception_design_v2.md`**: 에러 체이닝 구현 설계. 실제 코드 작성 시 이 문서의 구현 패턴을 따른다. (코드 구현용)

### 5.1 적용 예외 코드

| 예외 코드 | 발생 조건 | 처리 방침 |
|-----------|-----------|-----------|
| `{코드}` | {조건 요약} | {FATAL / PHASE_SKIP / ITEM_SKIP / FALLBACK / WARN} |

### 5.2 신규 예외 코드 제안 (해당 시에만 작성)

> `exception_spec_vN.md` §사용 규칙을 따른다.
> PM 리뷰 전까지 `(proposed)` 표식을 유지하고, 승인 후 `exception_spec_vN.md`에 병합한다.

```
### {코드} (proposed)
- 원인: {1~2줄 요약}
- context 필수: {키 목록}
- 방침: {FATAL / PHASE_SKIP / ITEM_SKIP / FALLBACK / WARN}
- 추가 근거: {기존 코드를 재사용하지 않는 이유}
```
